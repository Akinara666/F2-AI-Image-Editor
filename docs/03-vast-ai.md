# 03 — Vast.ai / готовый CUDA-контейнер (без Docker)

Когда сервер — это уже контейнер с CUDA и Python (Vast.ai, RunPod-pod и подобные),
backend запускается напрямую, без вложенного Docker. Вычислительная нагрузка
(backend) выполняется на сервере, лёгкий фронтенд — на клиентском компьютере.

- Сервер: [`deploy/run-vast.sh`](../deploy/run-vast.sh)
- Клиент: [`deploy/run-client.sh`](../deploy/run-client.sh)

---

## 0. Выбор инстанса

Используйте PyTorch- или CUDA-шаблон (не «чистый» Ubuntu) — тогда гарантированно
присутствуют Python, pip и драйвер/CUDA. torch может быть не предустановлен (зависит
от образа); скрипт самостоятельно загрузит колёса `cu128` при его отсутствии.

> Важно: «CUDA-шаблон» ≠ «PyTorch-шаблон». CUDA-образ даёт драйвер и toolkit, но
> torch в него может не входить — тогда `run-vast.sh` поставит его сам (это норма,
> не ошибка). `cu128` нужен для Blackwell/RTX 50xx; для старых карт можно передать
> `--torch-index https://download.pytorch.org/whl/cu121`.

Требования к инстансу:
- Python 3 и pip (на PyTorch/CUDA-шаблонах Vast.ai присутствуют);
- NVIDIA-драйвер (`nvidia-smi` работает).

На минимальном образе без pip предварительно выполните:
`apt update && apt install -y python3-pip python3-venv curl`.

## 1. Запуск на сервере (инстанс Vast.ai)

```bash
git clone https://github.com/Akinara666/F2-AI-Image-Editor.git
cd F2-AI-Image-Editor
bash deploy/run-vast.sh
```

Скрипт:
- устанавливает зависимости backend прямо в окружение инстанса (предустановленный
  CUDA-torch не затрагивается — определяется через `torch.cuda.is_available()`;
  колёса `cu128` загружаются только при необходимости);
- по умолчанию скачивает GGUF-модель Qwen для улучшения промпта и включает
  провайдер `qwen_gguf` (отключить — `--no-llm`); на GPU-инстансе сразу ставит
  `LLM_GPU_LAYERS=99`, чтобы Qwen работал на видеокарте, а не на CPU;
- **по умолчанию собирает фронтенд и отдаёт его тем же backend-ом** (один URL =
  сайт + API), так что напечатанный адрес — готовый редактор;
- запускает `uvicorn` на `0.0.0.0:8000`;
- по умолчанию поднимает Cloudflare quick-tunnel и выводит публичный адрес
  `https://<random>.trycloudflare.com`.

Первый запуск загружает torch, diffusers и модели (несколько ГБ); это занимает
несколько минут, которые тарифицируются. Повторные запуски быстрее.

> Запускай под `tmux` (`tmux new -s app`), иначе обрыв SSH остановит сервер.
> Отключиться не закрывая: `Ctrl+b`, затем `d`; вернуться — `tmux a`.

### Открыл URL — и сразу редактор (по умолчанию)

```bash
bash deploy/run-vast.sh
```

Скрипт собирает фронтенд и backend отдаёт его на том же адресе, что и API
(same-origin). Напечатанный `https://<random>.trycloudflare.com` — **готовый
сайт**: открыл в браузере и сразу редактор, без запуска `run-client.sh` на клиенте
и без CORS. Требует Node на инстансе (ставится автоматически при отсутствии).

### Только API (`--no-frontend`)

```bash
bash deploy/run-vast.sh --no-frontend
```

Фронт не собирается, публичный адрес — это API; редактор запускают на своём
компьютере через `run-client.sh` (см. раздел 2).

### Флаги `run-vast.sh`
| Флаг | Назначение |
|---|---|
| `--tunnel P` | провайдер публичного туннеля: `cloudflare` (по умолчанию), `ngrok`, `pinggy`, `localhost-run`, `none`. Подробнее — в разделе 2 |
| `--no-tunnel` | то же, что `--tunnel none` (доступ по нативному порту Vast.ai или SSH-пробросу) |
| `--ngrok-token T` | authtoken для `ngrok` (или переменная окружения `NGROK_AUTHTOKEN`) |
| `--no-venv` | установка в текущий python, а не в venv `deploy/.venv-vast` |
| `--optional` | установить xformers (llama-cpp-python и так ставится при включённом LLM) |
| `--no-frontend` (`--api-only`) | только API, без сборки фронта (фронт — на клиенте через `run-client.sh`) |
| `--with-frontend` | сборка фронта (по умолчанию уже включена; оставлен для совместимости) |
| `--no-llm` | не скачивать Qwen-GGUF и не включать провайдер `qwen_gguf` |
| `--llm-url U` | URL GGUF-модели (по умолчанию Qwen3-1.7B-Q8_0) |
| `--reinstall` | переустановить зависимости и перекачать GGUF |
| `--port N` | порт backend (по умолчанию 8000) |
| `--torch-index U` | индекс колёс torch (по умолчанию `cu128`; для старых карт — `cu121`) |
| `--no-follow` | не стримить логи backend в консоль (лог-файл пишется всегда) |
| `--new-tunnel` | поднять новый туннель, даже если живой уже есть (сменит URL) |
| `--stop` | остановить backend и туннель и выйти |
| `-h`, `--help` | справка |

> Правки из панели настроек (шестерёнка в UI) на vast пишутся в постоянный
> `deploy/backend.vast.env`. Чтобы применить их: `Ctrl+C` и снова `bash
> deploy/run-vast.sh` — backend перезапустится, **а публичный URL не сменится**
> (туннель живёт отдельно и переиспользуется). Токен для панели генерируется
> автоматически и печатается в баннере при запуске.

## 2. Публичный доступ — выбор способа

С настройками по умолчанию backend сам отдаёт собранный фронтенд, поэтому
презентационному компьютеру нужен только браузер и публичный адрес — запускать
`run-client.sh` не требуется. Остаётся выбрать, **как выставить сервер наружу**,
чтобы адрес открывался из той сети, откуда вы будете показывать проект. Способ
задаётся флагом `--tunnel`.

> **Если вы показываете проект из России.** Cloudflare (`trycloudflare.com`) у
> зрителя в РФ часто недоступен без VPN, поэтому вариант по умолчанию может не
> открыться. Это особенность сети, а не ошибка проекта. Для показа из России
> выберите другого провайдера (`--tunnel pinggy` или `--tunnel ngrok`) либо
> нативный порт (`--tunnel none`). Обязательно **проверьте заранее на том самом
> компьютере**, с которого будете презентовать, — фильтры в разных сетях отличаются.

| Провайдер (`--tunnel`) | Адрес | Аккаунт/токен | Особенности |
|---|---|---|---|
| `cloudflare` (по умолчанию) | `https://…trycloudflare.com` (443, TLS) | не нужен | проще всего; у зрителя в РФ часто недоступен без VPN |
| `pinggy` | `https://….pinggy.link` (443, TLS) | не нужен | работает по SSH, без установки; бесплатная сессия ~60 минут |
| `localhost-run` | `https://….lhr.life` (443, TLS) | не нужен | работает по SSH, без установки |
| `ngrok` | `https://….ngrok-free.app` (443, TLS) | нужен токен | стабильнее по сессии; токен бесплатно на ngrok.com |
| `none` | прямой `http://IP:порт` Vast.ai | не нужен | без TLS, нестандартный порт; строгий фильтр может не пропустить |

Все туннели поднимаются на стороне сервера и работают по 443/TLS; со стороны
зрителя важно лишь, пропустит ли его сеть конкретный домен. Именно это и стоит
проверить заранее.

### Туннель (`cloudflare` / `pinggy` / `localhost-run` / `ngrok`)

```bash
# на сервере (vast):
bash deploy/run-vast.sh --tunnel pinggy        # печатает https://….pinggy.link
```
Откройте напечатанный адрес в браузере — это готовый редактор (фронтенд и API на
одном URL). Для `ngrok` дополнительно укажите токен:
```bash
bash deploy/run-vast.sh --tunnel ngrok --ngrok-token <ВАШ_ТОКЕН>
# или: NGROK_AUTHTOKEN=<ВАШ_ТОКЕН> bash deploy/run-vast.sh --tunnel ngrok
```
Туннель запускается обособленно и переживает перезапуск backend: адрес не меняется
при повторном `run-vast.sh` (сменить намеренно — `--new-tunnel`, остановить —
`--stop`). Бесплатные туннели работают по принципу best-effort; у `pinggy` и
`localhost-run` бесплатная сессия ограничена по времени, поэтому для долгого показа
держите запасной канал.

> Если вы раздаёте только API (`--no-frontend`), фронтенд запускается на клиенте:
> `bash deploy/run-client.sh <НАПЕЧАТАННЫЙ_URL>`.

### Нативный порт Vast.ai (`--tunnel none`)

Прямой доступ по `http://IP:порт` без туннеля. Порт нужно открыть **при создании
инстанса** — в Docker-опциях шаблона добавьте:
```
-p 8000:8000
```
```bash
# на сервере (vast):
bash deploy/run-vast.sh --tunnel none
```
Если Vast.ai передал инстансу переменные `PUBLIC_IPADDR` и `VAST_TCP_PORT_8000`,
скрипт сам напечатает готовый адрес вида `http://PUBLIC_IP:34567`. Иначе найдите
соответствие `PUBLIC_IP:34567 -> 8000/tcp` в разделе «IP Port Info» инстанса.

Обратите внимание: это обычный HTTP (без TLS) на нестандартном порту. Сети,
пропускающие только 443/TLS (что типично для университетов), такой адрес, скорее
всего, заблокируют — поэтому проверяйте заранее и держите его как запасной вариант.

### Проброс порта по SSH (для личного использования)

Приватный доступ без публикации наружу: трафик шифруется, открывать порты не нужно.

```bash
# на сервере (vast):
bash deploy/run-vast.sh --tunnel none          # backend на 0.0.0.0:8000
```
```bash
# на клиенте, терминал 1 — проброс (host и порт SSH см. в разделе 3):
ssh -N -p 10882 root@ssh2281.vast.ai -L 8000:127.0.0.1:8000
#       порт SSH (не 8000)  хост Vast.ai      локальный_порт:цель_на_сервере
# не закрывайте этот терминал — пока он открыт, проброс активен
```
```bash
# на клиенте, терминал 2 — фронтенд на локальный конец проброса:
bash deploy/run-client.sh http://127.0.0.1:8000
# откройте http://localhost:5173
```
Схема прохождения запроса:
```
браузер -> localhost:5173 (фронтенд) -> localhost:8000 (клиент)
                                            |  SSH-проброс
                                            v
                                      127.0.0.1:8000 на сервере Vast.ai (backend)
```
Если локальный порт `8000` занят, выберите другой слева: `-L 8001:127.0.0.1:8000`,
и тогда `run-client.sh http://127.0.0.1:8001`. Этот способ требует SSH-клиента и
часто закрыт на чужих или ограниченных машинах, поэтому для показа с университетского
компьютера он, как правило, не подходит.

### Чек-лист перед презентацией

- За несколько дней **проверьте на том самом компьютере**, с которого будете
  показывать, какие адреса открываются. Узнать фильтр сети заранее иначе нельзя.
- Держите **два канала одновременно** (например, `pinggy` и `ngrok`): если один
  адрес заблокирован, вы сразу откроете другой.
- На случай полного отсутствия сети подготовьте **запись экрана** работающего
  приложения — это надёжная страховка.

## 3. Получение host и port для SSH (для проброса порта)

Помимо кнопки **Connect** в консоли Vast.ai:

vastai CLI:
```bash
pip install vastai
vastai set api-key <ваш_API_ключ>      # ключ из раздела Account, пункт API Key
vastai show instances                  # столбцы ssh_host / ssh_port
vastai ssh-url <INSTANCE_ID>           # -> ssh://root@ssh2281.vast.ai:10882
```

REST API:
```bash
curl -s https://console.vast.ai/api/v0/instances/ \
  -H "Authorization: Bearer <ваш_API_ключ>" \
  | jq -r '.instances[] | "ssh -p \(.ssh_port) root@\(.ssh_host)"'
```

Изнутри инстанса (при наличии Jupyter или веб-терминала):
```bash
echo "ssh -p $VAST_TCP_PORT_22 root@$PUBLIC_IPADDR"
```

Существует два режима SSH: proxy (`sshN.vast.ai`, доступен всегда) и direct
(`PUBLIC_IPADDR` и `VAST_TCP_PORT_22`, меньшая задержка). Для проброса `-L`
подходят оба.

## 4. Примечания по Vast.ai

- **Эфемерное хранилище.** Чтобы не загружать модели при каждой аренде, размещайте
  `backend/models` на persistent-томе (например, симлинк или монт на `/workspace`).
- **Публичный адрес без аутентификации** (любой туннель или нативный порт): любой
  обладатель ссылки сможет использовать ваш GPU. Для приватного доступа используйте
  проброс порта по SSH.
- **`Permission denied (publickey)`** при подключении по SSH — добавьте свой публичный
  ключ (`~/.ssh/id_ed25519.pub`) в аккаунт Vast.ai (раздел Account, пункт SSH Keys) или при
  создании инстанса.
- Переменные окружения backend — шаблон [`deploy/backend.env.example`](../deploy/backend.env.example).
  На vast скрипт создаёт из него рабочий файл `deploy/backend.vast.env` и backend
  читает/пишет именно его (через `ENV_FILE_PATH`); правки из панели настроек идут
  туда же. Никакого `backend/.env` на vast не создаётся. Справочник по ключам —
  в [инструкции 01](01-local.md#переменные-окружения).

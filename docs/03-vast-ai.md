# 03 — Vast.ai / готовый CUDA-контейнер (без Docker)

Когда сервер — это уже контейнер с CUDA и Python (Vast.ai, RunPod-pod и подобные),
backend запускается напрямую, без вложенного Docker. Вычислительная нагрузка
(backend) выполняется на сервере, лёгкий фронтенд — на клиентском компьютере.

- Сервер: [`deploy/run-vast.sh`](../deploy/run-vast.sh)
- Клиент: [`deploy/run-client.sh`](../deploy/run-client.sh)

---

## 0. Выбор инстанса

Используйте PyTorch- или CUDA-шаблон (не «чистый» Ubuntu) — тогда гарантированно
присутствуют Python, pip и драйвер/CUDA. torch может быть не предустановлен (зависит
от образа); скрипт самостоятельно загрузит колёса `cu128` при его отсутствии.

> Важно: «CUDA-шаблон» ≠ «PyTorch-шаблон». CUDA-образ даёт драйвер и toolkit, но
> torch в него может не входить — тогда `run-vast.sh` поставит его сам (это норма,
> не ошибка). `cu128` нужен для Blackwell/RTX 50xx; для старых карт можно передать
> `--torch-index https://download.pytorch.org/whl/cu121`.

Требования к инстансу:
- Python 3 и pip (на PyTorch/CUDA-шаблонах Vast.ai присутствуют);
- NVIDIA-драйвер (`nvidia-smi` работает).

На минимальном образе без pip предварительно выполните:
`apt update && apt install -y python3-pip python3-venv curl`.

## 1. Запуск на сервере (инстанс Vast.ai)

```bash
git clone https://github.com/Akinara666/F2-AI-Image-Editor.git
cd F2-AI-Image-Editor
bash deploy/run-vast.sh
```

Скрипт:
- устанавливает зависимости backend прямо в окружение инстанса (предустановленный
  CUDA-torch не затрагивается — определяется через `torch.cuda.is_available()`;
  колёса `cu128` загружаются только при необходимости);
- по умолчанию скачивает GGUF-модель Qwen для улучшения промпта и включает
  провайдер `qwen_gguf` (отключить — `--no-llm`);
- запускает `uvicorn` на `0.0.0.0:8000`;
- по умолчанию поднимает Cloudflare quick-tunnel и выводит публичный адрес
  `https://<random>.trycloudflare.com` — это адрес API.

Первый запуск загружает torch, diffusers и модели (несколько ГБ); это занимает
несколько минут, которые тарифицируются. Повторные запуски быстрее.

### Один URL = полноценный сайт (`--with-frontend`)

```bash
bash deploy/run-vast.sh --with-frontend
```

В этом режиме скрипт собирает фронтенд и backend отдаёт его на том же адресе, что и
API (same-origin). Тогда напечатанный `https://<random>.trycloudflare.com` —
**готовый сайт**: открыл в браузере и сразу редактор, без запуска `run-client.sh` на
клиенте и без CORS. Требует Node на инстансе (ставится автоматически при отсутствии).

### Флаги `run-vast.sh`
| Флаг | Назначение |
|---|---|
| `--no-tunnel` | без cloudflared (доступ по SSH-пробросу или нативному порту Vast.ai) |
| `--no-venv` | установка в текущий python, а не в venv `deploy/.venv-vast` |
| `--optional` | установить `requirements-optional` (xformers, llama-cpp-python) |
| `--with-frontend` | собрать фронт и отдавать его backend-ом (один публичный URL = сайт) |
| `--no-llm` | не скачивать Qwen-GGUF и не включать провайдер `qwen_gguf` |
| `--llm-url U` | URL GGUF-модели (по умолчанию Qwen3-1.7B-Q8_0) |
| `--reinstall` | переустановить зависимости и перекачать GGUF |
| `--port N` | порт backend (по умолчанию 8000) |
| `--torch-index U` | индекс колёс torch (по умолчанию `cu128`; для старых карт — `cu121`) |
| `-h`, `--help` | справка |

> Правки из панели настроек (шестерёнка в UI) на vast пишутся в постоянный
> `deploy/backend.vast.env` и применяются после повторного запуска `run-vast.sh`.

## 2. Подключение с клиента — три способа

Во всех вариантах фронтенд выполняется на вашем компьютере
([`run-client.sh`](../deploy/run-client.sh)); различается лишь способ, которым
браузер обращается к API. CORS для `localhost` разрешён по умолчанию.

### Вариант A — Cloudflare Tunnel (по умолчанию, простейший)
Открывать порты при создании инстанса не требуется (туннель использует исходящее
соединение).

```bash
# сервер:
bash deploy/run-vast.sh                       # выводит https://<random>.trycloudflare.com
# клиент:
bash deploy/run-client.sh https://<random>.trycloudflare.com
# откройте http://localhost:5173
```
Преимущества: не нужно открывать порты, HTTPS, работает за NAT.
Недостатки: адрес случайный и меняется при перезапуске; без аутентификации;
бесплатный туннель работает по принципу best-effort (возможны сбои при больших загрузках).

### Вариант B — проброс порта по SSH (без cloudflared; рекомендуется для личного использования)
Трафик шифруется, наружу ничего не публикуется, открывать порты не требуется.

```bash
# сервер:
bash deploy/run-vast.sh --no-tunnel           # backend на 0.0.0.0:8000
```
```bash
# клиент, терминал 1 — проброс (host и порт SSH см. в разделе 3):
ssh -N -p 10882 root@ssh2281.vast.ai -L 8000:127.0.0.1:8000
#       порт SSH (не 8000)  хост Vast.ai      локальный_порт:цель_на_сервере
# не закрывайте этот терминал — пока он открыт, проброс активен
```
```bash
# клиент, терминал 2 — фронтенд на локальный конец проброса:
bash deploy/run-client.sh http://127.0.0.1:8000
# откройте http://localhost:5173
```
Преимущества: приватно, с шифрованием, без публичной экспозиции.
Недостатки: необходимо держать SSH-сессию открытой.

Схема прохождения запроса:
```
браузер -> localhost:5173 (фронтенд) -> localhost:8000 (клиент)
                                            |  SSH-проброс
                                            v
                                      127.0.0.1:8000 на сервере Vast.ai (backend)
```
Если локальный порт `8000` занят, выберите другой слева: `-L 8001:127.0.0.1:8000`,
тогда `run-client.sh http://127.0.0.1:8001`.

### Вариант C — нативный порт Vast.ai (без cloudflared, прямой IP:порт)
Порт открывается только при создании инстанса — в Docker-опциях шаблона добавьте:
```
-p 8000:8000
```
```bash
# сервер:
bash deploy/run-vast.sh --no-tunnel
```
В разделе «IP Port Info» инстанса найдите соответствие вида `PUBLIC_IP:34567 -> 8000/tcp`.
```bash
# клиент:
bash deploy/run-client.sh http://PUBLIC_IP:34567
```
Преимущества: стабильный адрес на время аренды, без зависимости от Cloudflare.
Недостатки: порт нужно открыть заранее; соединение по обычному HTTP (без TLS),
публично доступно без аутентификации.

### Рекомендации по выбору
- Личное использование — вариант B (проброс по SSH): приватно, без публичного адреса.
- Быстрая демонстрация или проверка — вариант A (Cloudflare Tunnel).
- Необходим стабильный прямой адрес — вариант C.

## 3. Получение host и port для SSH (вариант B)

Помимо кнопки **Connect** в консоли Vast.ai:

vastai CLI:
```bash
pip install vastai
vastai set api-key <ваш_API_ключ>      # ключ из раздела Account, пункт API Key
vastai show instances                  # столбцы ssh_host / ssh_port
vastai ssh-url <INSTANCE_ID>           # -> ssh://root@ssh2281.vast.ai:10882
```

REST API:
```bash
curl -s https://console.vast.ai/api/v0/instances/ \
  -H "Authorization: Bearer <ваш_API_ключ>" \
  | jq -r '.instances[] | "ssh -p \(.ssh_port) root@\(.ssh_host)"'
```

Изнутри инстанса (при наличии Jupyter или веб-терминала):
```bash
echo "ssh -p $VAST_TCP_PORT_22 root@$PUBLIC_IPADDR"
```

Существует два режима SSH: proxy (`sshN.vast.ai`, доступен всегда) и direct
(`PUBLIC_IPADDR` и `VAST_TCP_PORT_22`, меньшая задержка). Для проброса `-L`
подходят оба.

## 4. Примечания по Vast.ai

- **Эфемерное хранилище.** Чтобы не загружать модели при каждой аренде, размещайте
  `backend/models` на persistent-томе (например, симлинк или монт на `/workspace`).
- **Публичный адрес без аутентификации** (варианты A и C): любой обладатель ссылки
  сможет использовать ваш GPU. Для приватного доступа используйте вариант B (SSH).
- **`Permission denied (publickey)`** при подключении по SSH — добавьте свой публичный
  ключ (`~/.ssh/id_ed25519.pub`) в аккаунт Vast.ai (раздел Account, пункт SSH Keys) или при
  создании инстанса.
- Переменные окружения backend — [`deploy/backend.env.example`](../deploy/backend.env.example);
  скрипт записывает их в `backend/.env` (читается `python-dotenv`). Справочник —
  в [инструкции 01](01-local.md#переменные-окружения).
